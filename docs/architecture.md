# 아키텍처 — 멀티에이전트 디자인 자동화 파이프라인

이 문서는 프로젝트의 설계 사상을 정리합니다. (회사/고객 데이터 없이 구조만 기술)

## 1. 한 줄 요약

> 디자인 제작을 **역할이 나뉜 여러 AI**에게 분담시키고, **단계 사이마다 검증 게이트**를 두어
> 결과물의 신뢰성을 보장하는 하네스 엔지니어링 프로젝트.

## 2. 왜 멀티에이전트인가

하나의 AI에게 `분석 → 기획 → 제작 → 검수`를 전부 맡기면:

1. 단계가 길어질수록 **맥락이 흐려져** 품질이 떨어진다.
2. AI가 근거 없는 내용을 사실처럼 만들어내는 **환각**이 발생한다.

그래서 단계를 쪼개 **각 단계를 가장 잘하는 AI**에게 맡기고, 사이에 검증을 끼운다.

## 3. 단계와 담당

| 단계 | 담당 AI | 강점 | 산출물(계약) |
|---|---|---|---|
| 분석·학습 | Antigravity (Gemini) | 시각자료 + 대용량 컨텍스트 | 관찰/요구사항 + claim 목록 |
| 기획 | Codex | 엄격·체계적 계획 | 기획안(YAML) |
| 제작 | Claude Code + MCP | 코딩·오케스트레이션·도구 제어 | .ai / SVG |
| 검증 | 검증 하네스 | 결정론적 점검 | verdict + score |

각 단계는 **정해진 형식의 입출력(계약)** 으로 연결된다. 예: 분석→제작 사이의
[학습표](../code/data-contract/learning_table_schema.md).

## 4. 환각을 막는 두 장치

### (1) 출처 라벨링 (claim labeling)
모든 주장에 `관찰 / 출처확인 / 추론 / 확인 필요 / 제안` 라벨을 강제한다.
근거가 필요한 라벨인데 출처가 없으면 사실로 확정되지 못한다.
→ 상세: [claim-labeling.md](claim-labeling.md), 구현: [`code/claim-verification/`](../code/claim-verification/)

### (2) 검증 게이트 (quality gate)
단계 사이에 자동 점검 스크립트(하네스)를 둔다. 통과하지 못한 산출물은 다음 단계로
넘어가지 못하고 이전 단계로 되돌아간다(재시도).
→ 구현: [`code/pipeline-orchestration/`](../code/pipeline-orchestration/)

## 5. 제작 단계의 무손실 원칙

디자인 요소를 추출할 때 **SVG 왕복을 금지**하고 네이티브 `.ai → .ai` 복제만 사용한다.
무손실 여부는 사람 눈이 아니라 **앵커(정점) 수 1:1 일치 + 래스터 0** 으로 결정론적으로 검증한다.
→ 구현: [`code/illustrator-automation/`](../code/illustrator-automation/)

## 6. 지식 축적·재학습

작업에서 쌓인 지식을 별도의 지식베이스(Obsidian)로 모아, 다음 분석 단계가 다시 학습한다.
파이프라인이 한 번 쓰고 끝나는 게 아니라 **시간이 지날수록 똑똑해지도록** 설계했다.

## 7. 이 저장소의 코드는 무엇인가

실제 운영 코드는 회사 자료가 얽혀 공개하지 않는다. 대신 위 각 장치의 **핵심 아이디어를
공개용으로 새로 작성한 실행 가능한 데모**를 [`code/`](../code/) 에 담았다. 모든 데모는
[`tests/`](../tests/) 로 검증되며 푸시마다 CI가 실행한다.
